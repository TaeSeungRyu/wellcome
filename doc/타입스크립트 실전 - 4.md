

## 🔥 11. “런타임 검증” 없으면 타입은 거짓말이다

TypeScript는 **컴파일 타임**만 보장한다.
API 응답은 언제든 깨진다.

```ts
const res = await fetch("/api/user");
const data: User = await res.json(); // ❌ 위험
```

👉 해결: **런타임 검증**

```ts
import { z } from "zod";

const UserDtoSchema = z.object({
  user_id: z.number(),
  user_name: z.string(),
});

type UserDto = z.infer<typeof UserDtoSchema>;

function parseUserDto(data: unknown): UserDto {
  return UserDtoSchema.parse(data);
}
```

👉 핵심

* “타입 = 신뢰” ❌
* “검증된 타입만 신뢰” ✅
* 장애 90% 여기서 줄어듦

## 🔥 12. “unknown > any” (진짜 중요)

```ts
function handle(data: any) { // ❌ 타입 포기
}
```

```ts
function handle(data: unknown) { // ✅ 안전 시작
  if (typeof data === "string") {
    // 여기서부터 안전
  }
}
```

👉 핵심

* `any` 쓰는 순간 타입 시스템 탈출
* `unknown`은 “검증 강제 장치”

## 🔥 13. Discriminated Union 제대로 쓰기

이미 AsyncState 잘 썼는데, 여기서 더 간다.

```ts
type Result<T> =
  | { type: "success"; data: T }
  | { type: "error"; error: ApiError };
```

```ts
function handle(result: Result<User>) {
  if (result.type === "success") {
    return result.data;
  }
  return result.error;
}
```

👉 핵심

* `if (result.error)` 이런 애매한 분기 ❌
* “케이스 기반 분기”로 버그 제거

## 🔥 14. exhaustive check (빠지면 컴파일 에러)

```ts
function assertNever(x: never): never {
  throw new Error("Unexpected value");
}

function render(state: AsyncState<User>) {
  switch (state.status) {
    case "idle":
    case "loading":
      return "loading...";
    case "success":
      return state.data.name;
    case "error":
      return state.error;
    default:
      return assertNever(state); // 🔥 핵심
  }
}
```

👉 핵심

* 상태 추가했는데 처리 안 하면?
  → 컴파일에서 바로 잡힘

## 🔥 15. “브랜드 타입”으로 실수 차단

```ts
type UserId = number & { readonly brand: unique symbol };
type PostId = number & { readonly brand: unique symbol };
```

👉 효과

```ts
function getUser(id: UserId) {}

getUser(123); // ❌ 실수 방지
```

👉 핵심

* 같은 number라도 의미 다르면 분리
* 대형 서비스에서 진짜 많이 씀

## 🔥 16. Generic은 “제약”이 핵심

```ts
function identity<T>(value: T): T {
  return value;
}
```

이건 장난 수준.

```ts
function getId<T extends { id: number }>(obj: T) {
  return obj.id;
}
```

👉 핵심

* Generic = 유연성 ❌
* Generic = “제약된 유연성” ✅

## 🔥 19. 함수 리턴 타입 “명시” (중요)

```ts
function getUser() {
  return { id: 1, name: "kim" };
}
```

👉 나중에 바뀌면 전파됨

```ts
function getUser(): User {
  return { id: 1, name: "kim" };
}
```

👉 핵심

* “의도 고정”
* 리팩토링 안전성 ↑

## 🔥 20. 타입 vs 인터페이스 기준

👉 간단 기준

* `type` → 유연한 조합 (union, mapped)
* `interface` → 객체 구조, 확장

👉 실무 팁

* 요즘은 **type 위주**가 많다 (더 강력함)

## 🔥 22. 테스트 가능한 타입 설계

```ts
// ❌ 테스트 어려움
function getUserProfile() {
  return fetch("/api/user");
}
```

```ts
// ✅ 의존성 분리
function getUserProfile(api: typeof getUser) {
  return api().then(...);
}
```

👉 핵심

* 타입 설계 = 테스트 설계
* DI(의존성 주입) 자연스럽게 연결됨

## 🔥 23. “에러 흐름”도 타입으로 통제

```ts
async function getUserProfile(): Promise<Result<User>> {
  try {
    const res = await getUser();
    return { type: "success", data: toUser(res.data) };
  } catch (e) {
    return { type: "error", error: { code: "500", message: "fail" } };
  }
}
```

👉 핵심

* throw 남발 ❌
* 흐름을 타입으로 통제
