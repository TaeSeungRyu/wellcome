

# 🔥 11. 타입 → 코드 생성 (Codegen 기반 설계)

지금까지는 “타입을 잘 쓰는 단계”였다면
여기부터는 **타입으로 코드를 생산한다**

### 👉 핵심 영역

* OpenAPI → TypeScript 자동 생성
* GraphQL Code Generator
* Zod → API mock / test 데이터 생성

```ts
type Routes = {
  "/user": {
    GET: { response: User };
  };
};
```

👉 확장

* API client 자동 생성
* mock server 자동 생성

👉 핵심
**“사람이 타입을 작성” → ❌
“타입이 코드를 만든다” → ✅**

# 🔥 12. 타입 안전한 상태 머신 (State Machine)

Discriminated Union 다음 단계

### 👉 추천 도구

* XState

### 👉 핵심 개념

* 상태 전이 자체를 모델링
* 이벤트 기반 설계

```ts
type Event =
  | { type: "FETCH" }
  | { type: "RESOLVE"; data: User };
```

👉 핵심
**“상태”가 아니라 “전이”를 통제한다**

# 🔥 13. Effect 시스템 (부작용 모델링)

여기부터 사고가 완전히 바뀐다.

### 👉 대표 라이브러리

* fp-ts
* Effect

### 👉 개념

* IO / Async / Error / Dependency 전부 타입으로 표현

```ts
Effect<Env, Error, Result>
```

👉 핵심
**try/catch / async/await → 추상화 대상이다**

# 🔥 14. 의존성 주입 (DI) + 타입

보통 DI는 런타임 개념인데
여기서는 **타입과 결합한다**

### 👉 핵심

* 서비스 인터페이스 타입 정의
* 구현 교체 가능

```ts
interface UserRepository {
  findById(id: number): Promise<User>;
}
```

👉 확장

* 테스트에서 mock 자동 교체
* effect system과 결합

👉 핵심
**“구현이 아니라 계약에 의존한다”**

# 🔥 18. 타입 디버깅 능력

이거 못 하면 고수 못 된다

### 👉 기술

* 타입 분해해서 보기
* intermediate type 만들기

```ts
type Debug<T> = { [K in keyof T]: T[K] };
```

👉 핵심
**타입 에러를 읽는 게 아니라 “해석”한다**
